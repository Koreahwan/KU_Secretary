# `/bot` UX 및 명령 정책

## 목적

이 문서는 자연어 비서 명령의 공식 제품 정책을 정의한다. 현재 저장소의 구현 기준 명령 목록은 `README.md`와 `docs/FEATURES_AND_IMPLEMENTATION_KO.md`를 따르며, 본 문서는 그 위에 `/bot`의 정식 UX 계약을 명시한다.

- 정식 자연어 비서 명령은 `/bot`이다.
- `/assistant`, `/asis`는 하위호환 alias로만 유지한다.
- 기존 고정 명령 `/today`, `/tomorrow`, `/weather`, `/plan`의 동작 계약은 유지한다.
- 이 문서는 정책 문서이며, 이번 티켓 범위에는 코드 변경이 포함되지 않는다.

## 1. 명령 위치와 역할

`/bot`은 자유형 자연어 요청의 공식 진입점이다. 사용자는 정해진 서브커맨드 문법을 외우지 않고도 조회나 실행 요청을 보낼 수 있어야 한다.

다만 `/bot`은 기존 고정 명령을 대체하지 않는다.

- 빠르고 결정적인 조회는 기존 고정 명령을 계속 지원한다.
- `/bot`은 고정 명령을 감싸는 상위 UX 레이어로 동작한다.
- 사용자가 같은 결과를 더 정확하게 얻을 수 있다면 `/today`, `/tomorrow`, `/weather`, `/plan` 같은 전용 명령을 안내할 수 있다.

## 2. Alias 정책

공식 문서, 온보딩, 도움말, 스크린샷, 예시, 봇 메뉴에는 `/bot`만 노출한다.

- `/assistant`와 `/asis`는 과거 사용자를 깨지 않기 위한 하위호환 alias다.
- alias는 새 기능 이름으로 취급하지 않는다.
- alias는 사용 예시의 기본값으로 쓰지 않는다.
- 향후 사용자 메시지, QA 체크리스트, 제품 문구는 `/bot` 기준으로 정렬한다.

## 3. 지원 범위

`/bot`이 다루는 사용자-facing 범위는 현재 제품 경계와 같아야 한다.

- 주 사용자 대상은 Korea University 학생이다.
- 일정, 수업, 과제, 공지, 날씨, 리마인더처럼 현재 저장소가 이미 다루는 학사 비서 범위 안에서만 동작을 약속한다.
- KU 중심 시간표/UClass/포털/Telegram 워크플로를 우선한다.
- 다른 학교에 대해 일반적인 Moodle 온보딩이 존재하더라도, `/bot` 문구가 KU 외 학교에 동일한 자동화 지원을 암시하면 안 된다.

조회형 지원 예시는 다음과 같다.

- `/bot 오늘 남은 수업이랑 마감 과제만 짧게 정리해줘`
- `/bot 내일 첫 수업 전에 확인할 공지랑 자료가 있는지 알려줘`
- `/bot 오늘 날씨랑 미세먼지 요약해줘`
- `/bot 지금 설정 상태에서 다음에 뭘 하면 되는지 알려줘`

실행형 지원 예시는 다음과 같다.

- `/bot 내일 오전 8시에 과제 제출 알림 잡아줘`
- `/bot inbox에 있는 일정 초안 보여주고 적용할지 물어봐`
- `/bot task 42를 완료 처리해줘`

실행형 요청은 기존 안전한 동작 단위로만 연결한다. 예를 들면 리마인더 생성은 `/plan`의 의미 체계를 재사용하고, 완료 처리나 inbox 반영은 기존 저장소의 명시적 명령 계약을 넘어서면 안 된다.

## 4. 비지원 범위

다음은 `/bot`의 약속 범위에 포함하지 않는다.

- 현재 저장소에 없는 새 자동화나 외부 서비스 조작
- KU 외 학교에 대한 동일 수준의 포털/시간표 자동화 약속
- DB 스키마 변경, 관리자용 운영 작업, 비밀정보 수정
- 사용자를 대신한 임의의 파괴적 변경
- 불명확한 요청을 추측으로 실행하는 동작

예를 들어 `/bot 수강신청 대신 해줘`, `/bot 모든 일정 다 지워`, `/bot 학교 지원 범위를 다른 대학까지 넓혀` 같은 요청은 지원 대상이 아니다.

## 5. 조회형과 실행형 UX 원칙

`/bot` 응답은 먼저 요청을 조회형과 실행형으로 구분해야 한다.

- 조회형: 저장된 데이터와 최근 sync 결과를 읽어 요약하거나 설명한다.
- 실행형: 리마인더 생성, inbox 반영, 완료 처리처럼 상태를 바꾸거나 후속 발송을 유발한다.

조회형 원칙:

- 읽기 전용 요청은 추가 확인 없이 바로 답한다.
- 이미 더 적합한 전용 명령이 있으면 짧게 함께 안내할 수 있다.
- 답변은 현재 저장소가 보장하는 데이터 원천 안에서만 말한다.

실행형 원칙:

- 기존에 정의된 안전한 실행 단위로만 변환한다.
- 필요한 시간, 대상, ID가 빠졌으면 먼저 보충 질문을 한다.
- 사용자가 기대하는 효과가 상태 변경인지 단순 조회인지 모호하면 실행 전에 의도를 확인한다.

## 6. 확인 질문 원칙

확인 질문은 "안전"과 "마찰 최소화"를 함께 만족해야 한다.

- 조회형 요청에는 확인 질문을 붙이지 않는다.
- 실행형 요청은 파라미터가 불충분하거나 해석이 둘 이상 가능할 때 먼저 한 번에 짧게 확인한다.
- 실행 결과가 저장, 적용, 완료 처리, 발송 예약처럼 상태를 바꾸면 사용자가 바꾸려는 대상이 맞는지 확인할 수 있어야 한다.
- 확인 질문은 한 번에 하나의 결정만 묻는다.
- 이미 충분히 명확한 단일 저위험 요청이라면 불필요한 재확인을 반복하지 않는다.

확인 질문 예시:

- `내일 오전 8시 기준으로 리마인더를 만들까요?`
- `inbox 초안 전체를 적용할까요, 아니면 특정 ID만 적용할까요?`
- `완료 처리할 대상이 task 42가 맞나요?`

## 7. 안전 원칙

`/bot`은 자연어 인터페이스여도 기존 명령보다 느슨하면 안 된다.

- 지원하지 않는 기능을 가능한 것처럼 말하지 않는다.
- 현재 데이터에 없는 사실을 추정해서 단정하지 않는다.
- 실행형 요청은 기존 저장소가 이미 가진 권한과 데이터 모델 안에서만 처리한다.
- 계정, 토큰, 비밀번호, 운영 설정 변경은 `/bot`의 일반 사용자 명령 범위에 포함하지 않는다.
- 고정 명령의 명확한 의미를 흐리지 않는다. `/bot`은 우회 진입점이지 별도 관리자 셸이 아니다.

## 8. 문서화 원칙

앞으로 자연어 비서 명령을 설명하는 문서는 다음 기준을 따른다.

- 첫 예시는 항상 `/bot`으로 쓴다.
- alias는 별도 하위호환 항목으로만 짧게 적는다.
- 조회형 예시와 실행형 예시를 함께 싣는다.
- `/today`, `/tomorrow`, `/weather`, `/plan` 같은 전용 명령의 안정적 사용처를 계속 명시한다.
